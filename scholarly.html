System Requirements Specification

Project: USA Accidents Insight - Web-based Accident Data Visualization Tool
Date: 24th of June 2025
Authors: Iordache George-Matei - Robert Zaharie

1. Introduction

1.1 Purpose

This document specifies the functional and non-functional requirements for the USA Accidents Insight web application. The system is designed to provide flexible visualization and analysis of USA accident data for user-defined time periods, including statistical analysis, cartographic rendering, and data export capabilities.

1.2 Scope

The USA Accidents Insight system will enable users to:

  • Visualize accident data through multiple chart types and interactive maps
  • Filter data by time period (years) and geographical location (state)
  • Export data and visualizations in CSV, WebP, and SVG formats
  • Perform multi-criteria searches of reported incidents
  • Add new accident records through an administrative interface

1.3 Product Overview

The system consists of a web-based dashboard for data visualization and analysis, supported by a PHP backend connected to an Oracle database. The application provides real-time statistical calculations and interactive mapping capabilities using modern web technologies.

2. System Overview

2.1 System Context

The USA Accidents Insight system operates as a standalone web application designed for academic and research purposes. It processes and visualizes historical accident data from the United States, providing insights through various statistical and geographical representations.

2.2 System Functions

  • Data filtering and querying based on temporal and geographical criteria
  • Statistical analysis and visualization generation
  • Interactive map rendering with accident location markers
  • Data export in multiple formats
  • Administrative data management

3. Functional Requirements

3.1 Data Visualization Requirements

FR-1: Dashboard Interface

Description: The system shall provide a web-based dashboard displaying accident statistics and visualizations.

Priority: High

Inputs: User filter selections (year range, state)

Outputs: Statistical charts, numerical summaries, interactive map

FR-2: Multiple Chart Types

Description: The system shall generate at least three different types of visualizations: total accidents bar chart, severity distribution chart, and time-of-day distribution chart.

Priority: High

Acceptance Criteria: Charts update dynamically based on filter selections

FR-3: Interactive Mapping

Description: The system shall display accident locations on an interactive map with severity-based markers.

Priority: High

Details: Map shall use marker clustering and display popup information for each accident

3.2 Data Filtering Requirements

FR-4: Temporal Filtering

Description: The system shall allow users to filter data by year range (2010-2022).

Priority: High

Validation: Start year must not exceed end year

FR-5: Geographical Filtering

Description: The system shall allow filtering by US state, including an "All States" option.

Priority: High

Implementation: Dropdown selection with all 50 US states

3.3 Data Export Requirements

FR-6: CSV Export

Description: The system shall export filtered accident data in CSV format.

Priority: Medium

Fields: All database fields including ID, location, severity, description, weather conditions, etc.

FR-7: Image Export

Description: The system shall generate and export statistical charts in SVG and WebP formats.

Priority: Medium

Content: Severity distribution chart with current filter parameters

3.4 Administrative Requirements

FR-8: Admin Interface

Description: The system shall provide an administrative interface for adding new accident records.

Priority: Medium

Fields: Date, coordinates, severity, state, description

FR-9: Data Validation

Description: The system shall validate all input data before database insertion.

Priority: High

Validation Rules: Date format, coordinate ranges, severity levels (1-4), valid state codes

4. Non-Functional Requirements

4.1 Performance Requirements

NFR-1: Response Time

Description: The system shall respond to user filter requests within 3 seconds under normal load.

Priority: Medium

NFR-2: Data Processing

Description: The system shall handle datasets with up to 100,000 accident records efficiently.

Priority: Medium

4.2 Usability Requirements

NFR-3: User Interface

Description: The system shall provide an intuitive web interface requiring no special training.

Priority: High

Standards: Responsive design supporting desktop and tablet devices

NFR-4: Accessibility

Description: The system shall follow basic web accessibility guidelines.

Priority: Low

4.3 Security Requirements

NFR-5: Input Sanitization

Description: The system shall sanitize all user inputs to prevent XSS and SQL injection attacks.

Priority: High

NFR-6: Data Validation

Description: The system shall validate all parameters server-side before database queries.

Priority: High

5. User Interface Requirements

5.1 Dashboard Interface

The main dashboard shall contain:

  • Filter controls (year range selectors, state dropdown, submit button)
  • Statistics display area with three main metric boxes
  • Chart visualization area below statistics
  • Interactive map section
  • Export buttons section

5.2 Administrative Interface

The admin panel shall provide:

  • Form fields for all required accident data
  • Input validation with error messaging
  • Success/failure feedback for data submission
  • Navigation back to main dashboard

6. System Architecture Requirements

6.1 Technology Stack

Component Technology Purpose
Frontend HTML5, CSS3, JavaScript User interface and client-side functionality
Backend PHP Server-side processing and database interaction
Database Oracle Database Data storage and retrieval
Mapping Leaflet.js Interactive map functionality
Charts Chart.js Statistical visualization

6.2 Database Requirements

The system shall use an Oracle database with the following key fields:

  • ID (Primary Key)
  • Start_Time (Date)
  • Start_Lat, Start_Lng (Coordinates)
  • Severity (1-4 scale)
  • State (2-letter code)
  • Description (Text)
  • Additional fields for comprehensive accident data

7. Constraints and Assumptions

7.1 Constraints

  • Data limited to years 2010-2022
  • Geographic scope limited to United States
  • Web browser compatibility required for modern browsers
  • Oracle database connectivity required

7.2 Assumptions

  • Users have basic computer literacy and web browser access
  • Accident data is pre-loaded into the database
  • Server environment supports PHP and Oracle connectivity
  • Internet connection available for map tile loading

8. Acceptance Criteria

8.1 Functional Acceptance

  • All filtering operations produce accurate results
  • Charts update correctly based on filter selections
  • Map displays accident locations with appropriate markers
  • Export functions generate files in specified formats
  • Admin interface successfully adds new records

8.2 Performance Acceptance

  • Dashboard loads within 5 seconds
  • Filter operations complete within 3 seconds
  • Map renders smoothly with up to 1000 markers
  • Export operations complete within 10 seconds

9. Future Enhancements

Potential future improvements may include:

  • User authentication and role-based access
  • Advanced filtering options (weather conditions, time of day)
  • Additional export formats (PDF, Excel)
  • Data visualization customization options
  • Mobile application development
  • Real-time data integration capabilities
